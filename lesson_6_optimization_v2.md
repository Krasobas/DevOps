# Оптимизация и профилирование Gradle [#505354]

Gradle — мощный инструмент сборки, который предоставляет множество механизмов для оптимизации производительности.

Когда я только начал работать с Gradle, то иногда возникало чувство, что он ничего не собрал, потому что пересборка проекта занимала миллисекунды.

Вот как можно ускорить сборку и анализировать её поведение.

## 1. Ускорение сборки с помощью Gradle Daemon

Gradle Daemon — это долгоживущий процесс, который запускается для выполнения сборки. Он позволяет избежать затрат на загрузку JVM и инициализацию проекта при каждой сборке. Gradle Daemon включён по умолчанию в современных версиях Gradle. Если он отключён, его можно активировать:

Командой:

```
./gradlew --daemon
```

Или добавлением строки в файл `gradle.properties` (в корневой папке проекта или в `~/.gradle/gradle.properties`):

```properties
org.gradle.daemon=true
```

Преимущества:

- Значительное ускорение сборки, особенно для крупных проектов.
- Меньшее время простоя для разработчиков.

Замечание: если процесс Daemon становится слишком «тяжёлым» (например, занимает много памяти), его можно остановить вручную:

```
./gradlew --stop
```

## 2. Кэширование сборок

Gradle поддерживает кэширование сборок (Build Cache), которое повторно использует результаты задач, если их входные данные не изменились. Добавьте в `gradle.properties`:

```properties
org.gradle.caching=true
```

Пример: если вы запускаете тесты, а код не менялся с последней сборки, результаты тестов будут взяты из кэша. Это особенно полезно для CI/CD, где одни и те же задачи выполняются на разных машинах.

Преимущества:

- Уменьшение времени сборки.
- Повышение производительности для задач, которые не зависят от изменений.

## 3. Инкрементальные сборки

Gradle выполняет только те задачи, которые необходимо обновить из-за изменений в проекте. Этот процесс называется инкрементальной сборкой.

**Как это работает:**

Gradle отслеживает изменения входных файлов и зависимостей каждой задачи. Если задача уже выполнена и её входные данные не изменились, она не будет выполняться повторно.

Если вы изменили только один файл в модуле, Gradle перекомпилирует только этот файл, а не весь модуль. Используйте флаг `--info`, чтобы увидеть, какие задачи выполняются:

```
./gradlew build --info
```

Чтобы использовать инкрементальные сборки, важно правильно настроить входные и выходные параметры задач, например с помощью аннотаций `@Input` и `@Output` в кастомных задачах.

## 4. Использование команды --profile для анализа производительности

Команда `--profile` генерирует HTML-отчёт о производительности сборки. Этот отчёт помогает понять, какие задачи занимают больше всего времени и где можно оптимизировать процесс.

Выполните сборку с параметром `--profile`:

```
./gradlew build --profile
```

**Где найти отчёт:** после выполнения команды отчёт будет сохранён в директории `build/reports/profile`.

**Что показывает отчёт:**

- Общее время сборки.
- Время выполнения задач: какие задачи выполнялись дольше всего.
- Инициализация и конфигурация: сколько времени ушло на инициализацию и конфигурацию Gradle.
- Операции ввода/вывода: как Gradle взаимодействовал с файловой системой.

**Пример использования:** если задача `test` занимает слишком много времени, отчёт поможет это выявить. Затем вы можете оптимизировать тесты, например, параллельным запуском.

## Рекомендации по оптимизации

1. **Включите Daemon и Build Cache** — это базовые настройки, которые обеспечивают значительное ускорение.
2. **Используйте инкрементальные сборки** — убедитесь, что входные и выходные параметры задач настроены корректно.
3. **Минимизируйте избыточные зависимости** — удалите неиспользуемые зависимости и модули.
4. **Параллельное выполнение задач** — добавьте в `gradle.properties`:

```properties
org.gradle.parallel=true
```

5. **Анализируйте производительность с --profile** — регулярно проверяйте, где можно сократить время выполнения задач.

## Задание

Откройте проект `job4j_devops`. Добавьте в проект: оптимизации сборки (Gradle Daemon, кэширование и параллельное выполнение), кастомную задачу, которая демонстрирует работу инкрементальной сборки на основе создания архива с ресурсами, и анализ производительности сборки с использованием профиля.

### Шаг 1. Включите оптимизации Gradle

Убедитесь, что в корневом файле `gradle.properties` (или создайте его) настроены следующие параметры:

```properties
# Включение Gradle Daemon
org.gradle.daemon=true

# Включение кэширования
org.gradle.caching=true

# Параллельное выполнение задач
org.gradle.parallel=true
```

> **Примечание:** параметр `org.gradle.configureondemand=true` является экспериментальным (incubating) и может вызывать проблемы в многомодульных проектах. Включайте его только если вы уверены в совместимости с вашим проектом.

Проверка — выполните сборку:

```
./gradlew build
```

Сравните время выполнения до и после включения оптимизаций.

### Шаг 2. Добавьте кастомную задачу для демонстрации инкрементальной сборки

Создайте задачу, которая архивирует содержимое директории `src/main/resources` в ZIP-файл. Задача должна поддерживать инкрементальную сборку: повторное выполнение задачи не должно происходить, если входные данные не изменились.

Добавьте эту задачу в ваш `build.gradle.kts`:

```kotlin
tasks.register<Zip>("archiveResources") {
    group = "custom optimization"
    description = "Archives the resources folder into a ZIP file"

    val inputDir = file("src/main/resources")
    val outputDir = layout.buildDirectory.dir("archives")

    inputs.dir(inputDir)
    outputs.file(outputDir.map { it.file("resources.zip") })

    from(inputDir)
    destinationDirectory.set(outputDir)
    archiveFileName.set("resources.zip")

    doLast {
        println("Resources archived successfully at ${outputDir.get().asFile.absolutePath}")
    }
}
```

### Шаг 3. Анализируйте производительность сборки

Запустите сборку с профилем:

```
./gradlew build --profile
```

После выполнения команды откройте отчёт, который находится в директории:

```
build/reports/profile/
```

**Вопросы для анализа:**

- Сколько времени заняли задачи `test`, `spotbugsMain` и новая задача `archiveResources`?
- Есть ли задачи, которые занимают больше времени, чем другие? Подумайте, как их можно оптимизировать.

### Проверка работы инкрементальной сборки

Выполните задачу:

```
./gradlew archiveResources
```

Убедитесь, что файл `resources.zip` создан.

Повторно выполните команду:

```
./gradlew archiveResources
```

Задача должна пропуститься, если директория `src/main/resources` не изменялась. В выводе вы увидите:

```
> Task :archiveResources UP-TO-DATE
```

Добавьте новый файл в `src/main/resources` и снова запустите задачу:

```
./gradlew archiveResources
```

Убедитесь, что задача выполняется заново, так как входные данные изменились.
