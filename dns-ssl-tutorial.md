# Домен, DNS и SSL-сертификаты: полное руководство

> **Стек:** домен на Cloudflare, сервер на Hetzner Cloud (Ubuntu 24.04)

---

## Как работает домен

Когда ты вводишь `krasobas.com` в браузере, происходит цепочка:

1. Браузер спрашивает у DNS-резолвера (обычно DNS провайдера интернета): «какой IP у krasobas.com?»
2. Резолвер идёт к корневым DNS-серверам → серверам зоны `.com` → серверам Cloudflare (ты указал их как nameservers при покупке домена)
3. Cloudflare отвечает: «krasobas.com → 168.119.xx.xx» (IP твоего Hetzner-сервера)
4. Браузер подключается к этому IP

Весь процесс занимает миллисекунды и кэшируется на каждом этапе. Вот почему изменения DNS не мгновенные — старый ответ закэширован у тысяч резолверов по миру. Но Cloudflare использует короткий TTL (время жизни кэша), поэтому изменения применяются за 1-5 минут.

---

## 1. IP сервера Hetzner

В **Hetzner Cloud Console** → твой сервер → раздел **Networking** — публичный IPv4.

Или на самом сервере:

```bash
curl ifconfig.me
```

## 2. Настройка DNS в Cloudflare

Зайди на [dash.cloudflare.com](https://dash.cloudflare.com) → выбери **krasobas.com** → **DNS** → **Records**.

Добавь две A-записи:

| Type | Name | Content | Proxy status | TTL |
|------|------|---------|--------------|-----|
| A | `@` | `твой IP` | DNS only (серая тучка) | Auto |
| A | `*` | `твой IP` | DNS only (серая тучка) | Auto |

### Что значат записи

**`@`** — алиас для самого домена `krasobas.com`. Когда кто-то вводит `krasobas.com` — DNS смотрит именно эту запись.

**`*`** — wildcard (подстановочный знак). Ловит **любой** поддомен, для которого нет отдельной записи: `jenkins.krasobas.com`, `grafana.krasobas.com`, `что-угодно.krasobas.com` — все пойдут на твой сервер. Без wildcard поддомены не будут резолвиться, и браузер покажет ошибку DNS.

### Почему DNS only, а не Proxied

Cloudflare предлагает два режима:

**Proxied (оранжевая тучка)** — весь трафик идёт через серверы Cloudflare. Они выступают прослойкой: DDoS-защита, CDN, WAF, свой SSL. Реальный IP сервера скрыт.

**DNS only (серая тучка)** — Cloudflare только отвечает на DNS-запросы. Трафик идёт напрямую на сервер.

Для работы с Traefik нужен **DNS only**. Причина: Traefik получает сертификаты через HTTP-01 challenge — Let's Encrypt делает запрос на порт 80 твоего сервера. Если трафик проксируется через Cloudflare, challenge может не пройти.

> **Что будет, если включить Proxied:** Cloudflare подставит свой SSL, и у тебя будет двойное шифрование (Cloudflare → Traefik). Это вызывает бесконечные редиректы и ошибку `ERR_TOO_MANY_REDIRECTS`. Решается настройкой SSL → Full (Strict) в Cloudflare, но это отдельная архитектура. Мы настраиваем Traefik как единственный терминатор SSL — это чище и прозрачнее.

> **Нюанс wildcard:** `*` работает только для поддоменов первого уровня. `jenkins.krasobas.com` — да. `ci.jenkins.krasobas.com` — нет, для вложенных нужна отдельная A-запись. На практике вложенные поддомены — редкость.

### Проверка

Подожди пару минут и проверь:

```bash
# Основной домен
dig krasobas.com +short

# Поддомен (благодаря wildcard)
dig jenkins.krasobas.com +short
```

Оба должны вернуть IP Hetzner-сервера.

> **Совет:** если видишь старый IP или пустой ответ, а запись точно правильная — исключи кэш провайдера, запросив через Google DNS:
> ```bash
> dig krasobas.com @8.8.8.8 +short
> ```

## 3. SSL/TLS: что это и зачем

TLS (все по привычке говорят «SSL», но правильное название — TLS) — протокол шифрования трафика между браузером и сервером. Без него данные летят открытым текстом, и любой узел на пути (Wi-Fi роутер, провайдер, промежуточный хоп) может их прочитать и изменить.

Сертификат — файл, который доказывает: «этот сервер действительно krasobas.com, а не кто-то, кто притворяется». Его выдаёт Certificate Authority (CA) — удостоверяющий центр.

**Let's Encrypt** — некоммерческий CA, выдающий сертификаты бесплатно и автоматически. До его появления в 2015 году сертификат стоил $10-100/год.

### Как Let's Encrypt проверяет владение доменом

Let's Encrypt не поверит на слово. Проверка через **challenge**:

**HTTP-01 challenge:** Let's Encrypt говорит: «положи файл с таким-то содержимым по адресу `http://krasobas.com/.well-known/acme-challenge/ТОКЕН`». Файл доступен — значит ты контролируешь сервер. Traefik делает это автоматически.

**DNS-01 challenge:** Let's Encrypt говорит: «создай TXT-запись `_acme-challenge.krasobas.com` со значением ТОКЕН». Запись появилась — значит ты контролируешь DNS. Этот способ нужен для **wildcard-сертификатов** (`*.krasobas.com` одним сертификатом).

> **Сертификаты Let's Encrypt живут 90 дней.** Это не баг — короткий срок снижает ущерб при утечке ключа. Автопродление обязательно. Traefik продлевает сам, за 30 дней до истечения.

> **Rate limits:** максимум 50 сертификатов на домен в неделю, 5 неудачных попыток в час. При первой настройке используй staging-сервер Let's Encrypt (выдаёт невалидные, но настоящие по структуре сертификаты, без лимитов). В Traefik это одна строка конфига — покажу во втором уроке.

## 4. Certbot: как это работает под капотом

Этот раздел — для понимания механики. В рабочей среде сертификатами будет управлять Traefik (второй урок), но знать, что происходит внутри, нужно для диагностики проблем.

### Установка

```bash
sudo apt update
sudo apt install -y certbot
```

### Получение сертификата (standalone)

Standalone — Certbot поднимает временный веб-сервер на порту 80 для HTTP-01 challenge. Порт 80 должен быть свободен:

```bash
# Проверь, не занят ли порт:
sudo ss -tlnp | grep :80

# Если занят — останови сервис:
sudo systemctl stop nginx  # или docker stop traefik
```

```bash
sudo certbot certonly --standalone -d krasobas.com
```

> **Почему `certonly`?** Без этого флага Certbot попытается установить сертификат в nginx/apache автоматически. `certonly` — только получает файлы, без вмешательства. Нам нужны файлы.

Certbot спросит email — укажи реальный. На него придёт предупреждение, если автопродление сломается.

### Структура файлов

```
/etc/letsencrypt/live/krasobas.com/
├── fullchain.pem   # сертификат + промежуточные CA (нужен серверам)
├── privkey.pem     # приватный ключ (секретный!)
├── cert.pem        # только твой сертификат
└── chain.pem       # только промежуточные
```

Большинству сервисов нужны `fullchain.pem` + `privkey.pem`.

> **Файлы в `live/` — символические ссылки** на реальные файлы в `/etc/letsencrypt/archive/`. При продлении ссылки обновляются автоматически. Не копируй их вручную — копии не обновятся.

> **`privkey.pem` — секретный файл.** Утечка = злоумышленник может выдавать себя за твой сайт (man-in-the-middle). Права `600`, владелец root. Certbot ставит это автоматически.

### Автопродление

Certbot создаёт systemd-таймер, который проверяет сертификаты дважды в день:

```bash
# Таймер активен?
sudo systemctl status certbot.timer

# Тестовый прогон:
sudo certbot renew --dry-run
```

> **Подводный камень:** если порт 80 постоянно занят (Traefik, nginx) — standalone не сможет продлить сертификат. Certbot молча упадёт, через 90 дней сертификат протухнет, сайт ляжет. Именно поэтому мы используем Traefik для управления сертификатами — он и трафик обслуживает, и challenge проходит одновременно, конфликта нет.

## 5. Wildcard-сертификат через DNS challenge

Обычный сертификат покрывает конкретные домены. Wildcard (`*.krasobas.com`) покрывает любой поддомен одним сертификатом.

Для wildcard обязателен DNS-01 challenge:

```bash
sudo certbot certonly --manual --preferred-challenges dns \
  -d krasobas.com -d "*.krasobas.com"
```

Certbot скажет создать TXT-запись. Иди в **Cloudflare Dashboard** → DNS → Records:

| Type | Name | Content |
|------|------|---------|
| TXT | `_acme-challenge` | значение от Certbot |

Подожди 1-2 минуты, нажми Enter.

> **`--manual` не поддерживает автопродление.** Через 90 дней придётся повторять руками. Для автоматического wildcard нужен Certbot с Cloudflare-плагином (certbot-dns-cloudflare) или Traefik с DNS challenge через Cloudflare API. Оба варианта — полноценные production-решения. Во втором уроке настроим через Traefik.

## 6. Диагностика

```bash
# Все сертификаты Certbot:
sudo certbot certificates

# Детали сертификата на работающем сервере:
echo | openssl s_client -connect krasobas.com:443 \
  -servername krasobas.com 2>/dev/null | \
  openssl x509 -noout -subject -dates -issuer

# Дней до истечения:
echo | openssl s_client -connect krasobas.com:443 \
  -servername krasobas.com 2>/dev/null | \
  openssl x509 -noout -enddate

# Удалить сертификат:
sudo certbot delete --cert-name krasobas.com

# Отозвать (если утёк ключ — делай немедленно):
sudo certbot revoke --cert-path /etc/letsencrypt/live/krasobas.com/cert.pem
```

> **Совет:** проверка в браузере — замочек в адресной строке → «Certificate is valid». Видно цепочку: твой сертификат → промежуточный CA → корневой CA.

## Итог: полный путь запроса к https://krasobas.com

```
Браузер
  │
  ├─ DNS-запрос к Cloudflare → получает IP Hetzner
  │
  ├─ TCP-соединение на порт 443
  │
  ├─ TLS-handshake: сервер предъявляет сертификат,
  │  браузер проверяет цепочку доверия до корневого CA
  │
  └─ Зашифрованный обмен данными
```

В рабочей среде сертификатами управляет Traefik — он получает, хранит и продлевает их без остановки сервисов и без конфликтов за порт 80. Certbot в этом случае не нужен. Но понимание механики (challenges, цепочки сертификатов, приватные ключи) — это то, что поможет тебе диагностировать проблемы, а не гуглить вслепую.
